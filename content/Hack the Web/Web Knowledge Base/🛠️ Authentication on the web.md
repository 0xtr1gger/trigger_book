## Authentication and authorization

>**Authentication**, or **AuthN**, is the process of **verifying the identity** of a user, device, or system to confirm they are genuinely who they claim to be.

- Or, more formally, per [`RFC 4949`](https://datatracker.ietf.org/doc/rfc4949/), authentication is:

>[!quote] The process of verifying a claim that a system entity or system resource has a certain attribute value. 
>

- Authentication consists of two basic stages:
	- **Identification** —  the process of claiming or stating one's identity.
	- **Verification** — the process of verifying the claimed to be genuine. 

>**Authorization**, or **AuthZ**, is the process of determining **what actions or resources** and *authenticated* user, device, or system is allowed to access. 

- Per [`RFC 4949`](https://datatracker.ietf.org/doc/rfc4949/), authorization is:

>[!quote] An approval that is granted to a system entity to access a system resource. 

Or

>[!quote] A process for granting approval to a system entity to access a system resource.

![](https://cdn.services-k8s.prod.aws.htb.systems/content/modules/269/auth_vs_auth.png)

>[!example]+ Airport analogy
>At an airport security checkpoint, showing your ID to confirm your identity is authentication. Once inside, showing your boarding pass to enter the plane is authorization.

>[!note] Authentication answers the question "Who are you?", and authorization answers the question "What are you allowed to do?".
## Authentication factors and common authentication methods

Authentication strength depends on *factors*.

>**Authentication factors** are categories of evidence used to verify a user's identity. 

There are three primary authentication factors:
- **Knowledge** — Something you **know** — Information the user remembers and enters.
	- Password
	- Passphrase
	- PIN code
	- Challenge-response (answer to a question or pattern)
	- Security question
- **Possession** — Something you **have** — Physical devices or tokens in the user's possession.
	- Hardware tokens (e.g., USB security keys, YubiKeys, smartcards, etc.).
	- One-Time Passwords (OTPs) (e.g., Microsoft Authenticator, Google Authenticator, Authy, etc.).
	- [TOTP](https://en.wikipedia.org/wiki/Time-based_one-time_password) (Time-based One-Time Password).
	- Out-of-band communication (e.g,. SMS codes or push notifications in cell phones or other devices).
- **Inheritance** — Something you **are** — Biometrics that identifies a user uniquely.
	- Fingerprint
	- Facial recognition
	- Iris scans
	- Retinal pattern
	- Voice recognition
	- DNA sequence
	- Signature

>[!note] Beyond the three traditional factors, some modern security frameworks recognize two more:
> 
> - **Location factors** — **Somewhere** you are — User's geolocation data.
> 	- Source IP address
> 	- Geolocation (GPS, WI-Fi, cellular data, or RFID)
> 
> - **Behavior factors** — Something you **do** — Behavior patterns that uniquely identify a user.
> 	- Keystroke dynamics
> 	- Mouse movement patterns
> 	- Gait analysis


>[!note] A **One-Time Password (OTP)** is a string of characters used to authenticate a user for a single login attempt or transaction.
>- OTPs are useless after single use, therefore attempts to steal an already used OTP is pointless.
>- OTPs are sent via an alternative communication channel, such as as an SMS, push notification, or hardware access token. 
>- OTPs are one of the most common type of authentication by ownership.

>[!note] **Out-of-band authentication** is an authentication method that uses a separate communication channel. 
>- For example, the user access the application via HTTP, but authenticates with an OTP sent to their mobile phone via SMS.

- Single-factor authentication relies on just one authentication method, such as only password authentication. 

>**Multi-Factor Authentication (MFA)** is a type of authentication that requires **two or more** **different authentication factors** to verify a user's identity.

- For example, MFA can use a password (knowledge factor) and a TOTP (Time-Based One-time Password) in an authenticator application on the user's smartphone (possession factor).

- If exactly two authentication factors are required, MFA is commonly referred to as **Two-Factor Authentication (2FA)**.

>[!info] Authentication that requires a password and security question, for example, is **not considered multi-factor**, because all authentication methods are of the same category (knowledge). Such authentication is called **multi-layer**.

>[!note]- OTPs vs. push notifications in authenticator applications
>- **OTPs** are time-based numeric codes, usually 6-8 digits, generated by an authenticator app (such as Google Authenticator, Authy, Microsoft Authenticator).
>	- The app generates a new OTP every 30-60 seconds based on a shared secret and time or event counter. The user manually reads the current OTP from the app and enters it into the login interface until it expires.
>- **Push-notifications** are real-time messages sent by the authentication server to the user's device via the authenticator app (e.g., Microsoft Authenticator, Duo Mobile).
>	- Once received the notification, the user can approve or deny authentication request. The process may also require entering a number displayed on the login page into the notification window.
>
>Push-notifications are more user-friendly and are considered more resistant to phishing since user consent is explicit.
>**Authentication**, or **AuthN**, is the process of verifying the identity of a user or client. 
## Attacks on authentication

- **Knowledge-based authentication**
	- Password-based attacks, such as:
		- Password brute-force
		- Password spraying
- **Ownership-based authentication**
	- Significantly more challenging, often involve physical attacks, such as theft or cloning of an object used for authentication (NFC cards, SIM cards, etc.).
- **Inheritance-based authentication**
	- Inheritance-authentication systems can be **irreversibly compromised** in the event of data breach, because users can not change their biometrics features, such as fingerprints, unlike passwords. 
	- See [`BioStar 2 breach`](https://www.vpnmentor.com/blog/cybersecurity/report-biostar2-leak/).

## Authentication protocols and mechanisms

>**Authentication protocols** and mechanisms define the rules and processes by which a system verifies the identity of users, devices, or services.

These protocols vary widely in complexity, security features, and use cases, from simple password transmission to robust multi-factor and federated identity systems.

Commonly used authentication mechanisms include:

- **Password-based authentication**
- **Token-based authentication**
- **OAuth 2.0**
- **OpenID Connect (OIDC)**
- **SAML (Security Assertion Markup Language)**
- **Kerberos**
- **Basic and digest HTTP authentication**
- **Passwordless authentication**
- **Other protocols**
### Password-based authentication

Password-based authentication is one of the most common mechanisms used to verify a user's identity.

The core idea is simple:
1. A user selects or is assigned a **username** (or user ID).
2. The user creates a **password**, which is a secret known only to them.
3. When the user attempts to log in, they provide their username and password.
4. The system verifies that the supplied password (or its hash) matches what it expects for that username.
5. If the password is correct, access is granted; otherwise, access is denied.

>[!important]
>Ideally, passwords are never stored in plain-text.
>- The password is **hashed** using a cryptographic hash algorithm, such as SHA-2, SHA-3,  bcrypt, Argon2, or PBKDF2.
>- A **salt** — random data unique per user — should be added to the password before hashing to prevent pre-computed hash attacks (rainbow tables).
>**The system stores the resulting salted hash and the salt, not the plaintext password.**

>[!note] On security of password-based authentication
> - Security of password-based authentication depends on the password strength: weak passwords can easily be guessed or cracked.
> - If the password is reused multiple times across different platforms, a breach in one system can lead to a compromise in others.
> - Passwords should only be sent over an encrypted communication channel (TLS/HTTPS) to prevent eavesdropping.
> - Since passwords can be forgotten, there should be password reset functionality. Incorrect implementation introduces vulnerabilities and becomes a weak security point.
> - Passwords alone are usually not enough. MFA is often recommended.

See [[྾_password-based_authentication_vulnerabilities]]

Common password-based protocols and implementations:

- **Standard web login forms**
	- Username and password fields; submitted to the server for verification.

- **HTTP Basic Authentication**
	 - Sends Base64-encoded credentials in HTTP headers; insecure without HTTPS.

- **PAP (Password Authentication Protocol**
	- Used in PPP (Point-to-Point Protocol) authentication.
	- Sends passwords in plain-text, insecure, considered weak.

#### HTTP Basic Authentication

>In **HTTP Basic Authentication**, the client sends a username and password in each HTTP request's `Authorization` header.

Credentials are concatenated as `username:password` and then Base64-encoded (not encrypted):

```HTTP
Authorization: Basic Base64(username:password)
```

>[!important] Base64 encoding is **NOT encryption**; it merely encodes data (easy to decode).

- The server decodes the Base64 string, extracts username and password, and verifies them.

- If verification fails, the server returns a `401 Unauthorized` HTTP status code with a `WWW-Authenticate: Basic` header with optional `realm` parameter to prompt the client to resend credentials.

```HTTP
WWW-Authenticate: Basic realm="User profile"
```

>[!important] Security considerations of HTTP Basic Authentication
>- Credentials are sent with _every_ request; this means **using HTTPS/TLS is mandatory** to prevent interception (MitM, Man-in-the-Middle attacks).
>- Browsers typically cache and auto-fill credentials after they've been entered once.
>- HTTP Basic Authentication is simple to implement; supported by all HTTP clients.
>- No session state is needed on the server; each request includes full credentials.
>- **No protection against replay attacks or brute-force.**
>- No other way to log out other than closing the browser or cleaning browser cache manually.
>- Vulnerable to **CSRF (Cross-Site Request Forgery)**; see [[🖉CSRF]].
>- No built-in support for MFA or hashing.
>- Mostly used in simple internal or legacy systems in trusted network environments.

>[!example]- Example: HTTP Basic Authentication in Flask (Python)
>See how simple it is to implement Basic Authentication in Flask:
>
> ```Python
> from flask import Flask
> from flask_httpauth import HTTPBasicAuth
> from werkzeug.security import generate_password_hash, check_password_hash
> 
> app = Flask(__name__)
> auth = HTTPBasicAuth()
> 
> users = {
>     "username": generate_password_hash("password"),
> }
> 
> @auth.verify_password
> def verify_password(username, password):
>     if username in users and check_password_hash(users.get("username"), password):
>         return username
> 
> @app.route("/")
> @auth.login_required
> def index():
>     return f"You have successfully logged in, {auth.current_user()}"
> 
> if __name__ == "__main__":
>     app.run()
> ```
#### HTTP Digest Authentication - Nonce mechanism prevents replay attacks.
> - Password is hashed and never sent in plain text, but it's still vulnerable if MD5 hash algorithm is used (it's been cracked long time ago).
> - Vulnerable to MitM if nonces or secrets are predictable.
> - More complex to implement than Basic.
> - Logout is as awkward as in Basic Authentication.
> - Susceptible to downgrade and session fixation attacks.

>**HTTP Digest Authentication** ([RFC 7616](https://datatracker.ietf.org/doc/html/rfc7616)) is a more complex protocol designed to address weaknesses of HTTP Basic Authentication.

Instead of sending credentials, the client sends a **MD5-hashed** form of the credentials combined with a server-supplied nonce (number used once).

> [!important]+ HTTP Digest Authentication process
> 
> 1. The client requests a resource that requires authentication. 
> 
> ```HTTP
> GET /secure-area HTTP/1.1
> Host: example.com
> ```
> 
> 2. The server responds with `401 Unauthorized` and sends a `WWW-Authenticate: Digest` header that contains a **realm** (a namespace for username/password) and a **nonce** (a random value used to prevent replay attacks) and other parameters.
> 
> ```HTTP
> HTTP/1.1 401 Unauthorized
> WWW-Authenticate: Digest realm="REALM", nonce="NONCE", algorithm=MD5, qop="auth"
> ```
> 
> 3. The client computes response hash:
> 
> ```
> hash = MD5(MD5(USERNAME:REALM:PASSWORD):nonce:MD5(METHOD:URI))
> ```
> 
> 4. The client sends an `Authorization` header:
> 
> ```HTTP
> Authorization: Digest username="USERNAME", realm="REALM", nonce="NONCE", uri="/path", response="hash", ...
> ```
> 
> ```HTTP
> GET /secure-area HTTP/1.1
> Authorization: Digest username="USERNAME", realm="REALM", nonce="NONCE", uri="/secure-area",
> response="COMPUTED_MD5_HASH", qop=auth, nc=00000001, cnonce="CNONCE"
> ```
> 
> 5. The server calculates the expected hash and compares it to the one sent by the client. If there's a match, access is granted, otherwise denied.

>[!important] Security considerations of HTTP Basic Authentication
>- Widely supported by browsers but less common than Basic.
> - Nonce mechanism prevents replay attacks.
> - Password is hashed and never sent in plain text, but it's still vulnerable if MD5 hash algorithm is used (it's been cracked long time ago).
> - Vulnerable to MitM if nonces or secrets are predictable.
> - More complex to implement than Basic.
> - Logout is as awkward as in Basic Authentication.
> - Susceptible to downgrade and session fixation attacks.
### Token-based authentication

>In **token-based authentications**, **after a user’s identity is verified once (e.g., via username and password)**, the system issues a **token** — a digitally encoded and cryptographically signed piece of information that certifies a user's identity.
>The client then presents this token, usually in an HTTP header, in each subsequent request for access to protected resources, in place of resubmitting credentials every time.

>[!important] The **token "represents" the user’s identity and/or authorization rights** for a limited period of time.

- Tokens may also carry *claims* or *attributes* about the user, such as roles, permissions, and other details, which can be interpreted by the server.

>[!important] **Stetelessness**
> - Tokens are **stateless**: the server doesn't keep the user's session state, as the token itself carries all necessary information. 
> - This makes token-based authentication very scalable, especially in distributed systems and microservices architectures.

The workflow typically works like this:

1. **User authentication (login)**
	- The user authenticates to the application (such as providing their username and password).

2. **Token generation**
	- Upon successful verification, the server generates a **token**. 
	- The token encodes information about the user identity and may include other details, such as permissions the user has or the roles they're assigned.
	- One of the most commonly used token standards is **JWT** (JSON Web Token). The **token “represents” the user’s identity and/or authorizations** for a limited duration.
	- The token is sent to the client (browser, mobile application), and the client stores the token (such as in-memory, in local storage, or as a cookie)

3. **Access to protected resources**
	- In each subsequent request to protected endpoints, the client includes the token, typically in the HTTP `Authorization` header as a `Bearer` token.
	- The server validates the token's authenticity, checking signature, integrity, expiration, and any revocation lists or blacklists.
	- If the token is valid, the server authorizes the request according to the permissions of the user whom this token is tied to. Otherwise, access is denied.

4. **Token expiration and renewal**
	- Tokens should always have an expiration time. This limits the time frame when a stolen token is useful.
	- When a token expires, the client may need to re-authenticate or renew the token.

```HTTP
Authorization: Bearer <JWT>
```

See [[྾_how_JWT_works]] and [[྾_JWT]]

>[!important] Bearer tokens 
>**A bearer token** is a cryptic, usually opaque string issued by an authentication server after a user successfully logs in. There're called "bearer" tokens because **any party ("bearer") who possesses the token is granted access to the associated resources with no need for further identity verification or proof of ownership**.
>Essentially, possession alone is sufficient — it's like carrying a physical key that unlocks a door; without any additional checks.

>[!important] If an attacker steals a bearer token, they get a full access to the account the token is associated with for the duration of the session.

---

> [!important]+ Weak points and potential vulnerabilities of token-based authentication
> 
> - **Token theft**
> 	- **MitM:** tokens (especially bearer tokens) sent over HTTP or compromised HTTPS can be intercepted and reused by attackers until expiry.
> 	- **XSS:** If tokens are stored in the browser (e.g., `localStorage` or `sessionStorage`), [[྾_XSS]] vulnerabilities can leak tokens.
> 	- **Device compromise:** stolen devices can give full access to tokens stored on physical storage.
> 	- **Token leakage:** tokens can also be leaked through URLs, logs, or other sources.
> 
> - **Lack of central revocation**
> 	- Stateless JWTs are valid until expiry, and immediate revocation is not supported unless additional infrastructure, such as token blacklists or short expiry with refresh tokens, is implemented.
> 	- **Refresh token abuse:** In turn, if not properly implemented, refresh tokens can be abused to keep sessions alive indefinitely.
> 
> - **Vulnerabilities in token implementation**
> 	- **Algorithm confusion attack**: if the signature algorithm isn't strictly enforced or is weak, attackers can forge tokens.
> 	- Over-scoped or long-lived tokens can make an application vulnerable t privilege escalation.
### OAuth 2.0

>OAuth 2.0 is an industry-standard **authorization framework** designed to allow third-party applications to obtain limited access to a user's resources on an HTTP service, **without sharing user credentials**.

In OAuth, the client application is issued access tokens that can be used to access protected resources on behalf of the user. These access tokens are usually **bearer tokens** (often JWT).

>[!important] OAuth 2.0 is **not an authentication protocol** but an authorization framework; however, it is often used alongside **OpenID Connect** to provide authentication.

Essentially, OAuth 2.0 with OIDC is what you use to login to websites via GitHub, Google, Facebook, etcetera.

For more information, see [[྾_how_OAuth_works]] and [[྾_OAuth]].
### OpenID Connect (OIDC)

>**OpenID Connect (OIDC)** is an **authentication protocol** built on top of OAuth 2.0 authorization framework. OIDC allows client applications to **authenticate users** in a standardized way.

See [[྾_OIDC]].

### SAML (Security Assertion Markup Language)

>**SAML (Security Assertion Markup Language)** is an XML-based, open-standard protocol used for exchanging authentication and authorization data between parties, particularly **Identity Provider (IdP)** and **Service Provider (SP)**.

SAML powers single sign-on (SSO) for enterprise, web apps, and federated services.

See [[྾_SAML]].

### Kerberos

>**Kerberos** is a **network authentication protocol** designed to verify user and service identities by using a trusted third-party system and cryptographic tickets rather than sending passwords over the network.

Kerberos is widely used in **enterprise environments**. Notably, it's integrated into Microsoft Windows domains via Active Directory/Entra ID.

To authenticate users and services, Kerberos relies on **symmetric key cryptography** and **trusted Key Distribution Center (KDC)**.

See [[Kerberos]].
### Passwordless authentication

>**Passwordless authentication** is a method of verifying a user's identity without the use of traditional passwords.

Instead of passwords, passwordless authentication uses alternative methods such as biometrics, hardware keys, or one-time codes to confirm identity. It's used to minimize common attack vectors related to passwords.

The process looks like this:

1. **User initiation**
	- The user initiates a login by providing their username, email, or other ID.
2. **Identity verification** 
	- The system verifies the user's identity using one or more possession or inheritance factors, such as:
		- **Biometrics:** fingerprint scan, facial recognition, iris scan, etc.
		- **One-time passcodes (OTPs):** send via SMS, email, or authenticator applications.
		- **Hardware tokens:** physical devices like YubiKeys or security keys that support protocols like FIDO2/WebAuthn.
		- **Push-based notification:** user receives a prompt on a trusted device to approve or deny authentication.
3. **Access granted**
	- Once verified, the user gains access. No passwords needed.


>[!note] OTPs vs. push notifications in authenticator applications
>- **OTPs** are time-based numeric codes, usually 6-8 digits, generated by an authenticator app (such as Google Authenticator, Authy, Microsoft Authenticator).
>	- The app generates a new OTP every 30-60 seconds based on a shared secret and time or event counter. The user manually reads the current OTP from the app and enters it into the login interface until it expires.
>- **Push-notifications** are real-time messages sent by the authentication server to the user's device via the authenticator app (e.g., Microsoft Authenticator, Duo Mobile).
>	- Once received the notification, the user can approve or deny authentication request. The process may also require entering a number displayed on the login page into the notification window.
>
>Push-notifications are more user-friendly and are considered more resistant to phishing since user consent is explicit.

Implementation examples:
- Windows Hello and Apple Face ID use facial recognition
- Fingerprint scanners on smartphones
- Magic links — a user receives a login link via email; clicking the link logs them in automatically
- OTPs generated by applications like Microsoft Authenticator, Google Authenticator, Authy, etc.

The benefits are obvious: no risk for password theft, reuse, brute-force, phishing — nothing to steal; no need to memorize those passwords or store them in password managers. Just one click on your phone or a gaze into the camera — and your're in.

What are the risks?

> [!important]+ Weak points and potential vulnerabilities of passwordless authentication
> 
> - **Device theft or loss**
> 	- If a device used for authentication is lost or stolen, attackers can gain full access to user's accounts.
> - Biometric spoofing
> 	- Attackers can fake fingerprints, use 3D-printed faces, deepfakes, or recorded voice samples to bypass verification. Only advanced liveness detection and multi-modal biometrics can help, but it's not foolproof and often requires expensive scanners.
> - **SIM card swapping**
> 	- SMS-based tokens are vulnerable to SIM swap fraud or interception on insecure networks.
> - **Weaknesses in account recovery mechanisms**
> 	- If passwordless systems rely on traditional fallback methods (e.g., password reset via email), attackers may exploit these vectors.
### Other protocols

Other commonly used authentication protocols include:

- **LDAP (Lightweight Directory Access Protocol)**
	- Used primarily for querying and authenticating against directory services like Active Directory.
	- See [[LDAP]]
- **RADIUS (Remote Authentication Dial-In User Service)**
	- Network protocol for centralized Authentication, Authorization, and Accounting (AAA).
	- See [[RADIUS]]
- **CHAP (Challenge Handshake Authentication Protocol)**
	- Provides stronger authentication than PAP by using challenge-response mechanisms.
- **EAP (Extensible Authentication Protocol)**
	- Framework supporting multiple authentication methods, including for wireless networks.
